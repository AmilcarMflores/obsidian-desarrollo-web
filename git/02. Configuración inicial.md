## Instalación
### Linux
En este ejemplo, lo haremos para Ubuntu:

En la terminal, ejecutamos este comando:
```bash
apt-get install git
```
o
```bash
add-apt-repository ppa:git-core/ppa
apt update; apt install git
```
### Windows
Es descargar el archivo .exe e instalarlo (en la instalación te da varias opciones pero dejalo como está).
## Configurando Git por primera vez
Git tiene el comando `git config` que nos permite establecer variables de configuración.

Estas variables pueden almacenarse en tres lugares:
1. Archivo `/etc/gitconfig`: que son para todos los usuarios del sistema y para establecer en este usaremos el comando `git config --system`.
2. Archivo `~/.gitconfig` o `~/.config/git/config`: que son para usuarios específicos y para establecer en este usaremos `git config --global`.
3. Archivo `config` que viene en el directorio de Git (`.git/config`): de la carpeta que trabajamos (configuración solo a esa carpeta) y para establecer usaremos `git config --local`.

Las configuraciones en `.git/config` tiene preferencia sobre las de `/etc/gitconfig`. 

----
**Lo primero que debes hacer al instalar Git es establecer el nombre de usuario y dirección de correo electrónico.** Esto es importante porque los *commits* de Git utilizan esta información para identificar al autor de los cambios:
```bash
git config --global user.name "John Doe"
git config --global user.email johndoe@example.com
```
Para ver nuestra configuración:
```bash
# Muestra todas las propiedades que Git ha configurado
git config --list
```
### Finales de línea
La configuración `core.autocrlf` sirve para gestionar automáticamente los finales de línea de los archivos de texto cuando trabajamos en diferentes sistemas operativos.

Los sistemas operativos no se ponen de acuerdo en cómo marcar el final de una línea:
- **Windows**: Usa dos caracteres, *Carriage Return + Line Feed* (`CRLF`).
- **Linux y macOS**: Usan solo uno, *Line Feed* (`LF`).

**Si no configuras esto, al trabajar en equipo, Git podría detectar que todo el archivo ha cambiado solo porque los saltos de línea son diferentes, lo que arruina el historial de cambios y genera conflictos innecesarios**.

Entonces en **Windows** el comando es `git config --global core.autocrlf true`. Mientras que en **Linux y macOS** es `git config --global core.autocrlf input`.
### .gitattributes
Otra manera de configurar los finales de línea es creando este archivo, nos garantiza que todos los desarrolladores (Windows, Mac o Linux) trabajen con las mismas reglas automáticamente.

**Implementación:**
1. Crear el archivo: En la raíz del proyecto, creamos un archivo llamado `.gitattributes`
2. Copia y pega este contenido, que es el estándar moderno para la mayoría de proyectos web y de software:
```bash
# 1. Tratar todos los archivos como LF al confirmar (normalización)
* text=auto

# 2. Forzar que los archivos de texto siempre se conviertan a LF al subir
# y se adapten al sistema operativo del usuario al descargar
*.js text
*.html text
*.css text
*.json text
*.md text

# 3. Forzar que ciertos archivos SIEMPRE sean LF (útil para scripts de Bash)
*.sh text eol=lf

# 4. Forzar que ciertos archivos SIEMPRE sean CRLF (ej. archivos de configuración de Windows)
*.bat text eol=crlf

# 5. Marcar archivos como binarios (no se tocan los finales de línea)
*.png binary
*.jpg binary
*.pdf binary
```
3.  Aplicar los cambios a un proyecto existente-. Si ya tienes archivos en tu repositorio, crear el `.gitattributes` no los cambiará de inmediato. Debes "refrescar" el índice de Git con estos comandos:
```bash
# Borra el índice de Git (no borra tus archivos físicos)
git rm --cached -r .

# Vuelve a agregar todo (Git aplicará las nuevas reglas del .gitattributes)
git add .

# Crea el commit de normalización
git commit -m "Configuración de .gitattributes y normalización de finales de línea"
```
**Al usar este archivo, ya no importa qué valor tenga el usuario en su git config --global core.autocrlf. El archivo .gitattributes tiene prioridad y evita el famoso error: "LF will be replaced by CRLF".**
