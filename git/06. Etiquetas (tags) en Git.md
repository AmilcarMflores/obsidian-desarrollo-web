Una etiqueta es un **punto de referencia inmutable** que apunta a un momento específico en la historia de tu repositorio.

A diferencia de las ramas, que se desplazan automáticamente cuando añades nuevos commits, las etiquetas permanecen fijas en el commit donde fueron creadas.
## ¿Por qué son importantes?
- **Hitos de lanzamiento**: Permiten marcar versiones oficiales (ej. `v1.0.0`, `v2.4.0`) sin que se pierdan entre miles de commits.
- **Estabilidad**: Garantizan que cualquier desarrollador pueda volver exactamente al mismo estado del código que se envió a producción.
- **Automatización**: Muchas herramientas de despliegue automático (CI/CD) se activan solo cuando detectan que se ha subido una nueva etiqueta al servidor.
## Listar etiquetas
Para ver qué versiones existen en tu proyecto:
```bash
git tag                # Listado alfabético
git tag -l "v1.8.5*"   # Filtrar etiquetas de una serie específica
```
## Crear etiquetas
Existen dos tipos principales, aunque se recomienda siempre el uso de **anotadas**:
- **Etiquetas anotadas**: Almacenan metadatos (autor, fecha, mensaje) y son objetos completos de Git.
```bash
git tag -a v1.4 -m "Versión estable lista para producción"
```
- **Etiquetas ligeras**: Son solo un puntero o *alias* al commit, sin información extra.
```bash
git tag v1.4-lw
```
Para ver la información de una etiqueta usa: `git show v1.4`
## Etiquetado tardío
Puedes etiquetar un commit del pasado si olvidaste hacerlo en su momento. Solo necesitas el identificador *(checksum)* del commit:
```bash
git tag -a v1.2 9fceb02 -m "Etiquetando versión antigua"
```
## Compartir etiquetas (*push*)
Por seguridad, Git no sube las etiquetas al servidor de forma automática al hacer `push`. Debes enviarlas explícitamente:
- Subir una específica: `git push origin v1.5`
- Subir todas las locales: `git push origin --tags`
